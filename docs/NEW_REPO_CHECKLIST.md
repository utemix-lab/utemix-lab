# Чеклист создания нового репозитория

Перед созданием нового репозитория в экосистеме utemix-lab, убедитесь, что он соответствует требованиям согласованности.

---

## 1. Определение роли

### Вопросы перед созданием

1. **Зачем нужен этот репо?**
   - Какую проблему решает?
   - Почему нельзя решить в существующих репо?

2. **К какому слою относится?**
   - Философский (редко — только фундаментальные принципы)
   - Онтологический (определяет сущности)
   - Навигационный (определяет маршруты)
   - Контрактный (определяет представление)
   - Рендер (отображает данные)
   - Исследовательский (генерирует идеи)
   - Производный (расширяет принципы)

3. **Как связан с другими репо?**
   - От кого получает данные?
   - Кому отдаёт данные?
   - Какие репо зависят от него?

## 2. Обязательные файлы

### Минимальный набор

```
new-repo/
├── README.md           # С секцией "Роль в экосистеме"
├── LICENSE             # MIT (копия из любого репо)
└── .gitignore          # Стандартный для типа проекта
```

### README.md — обязательные секции

```markdown
# repo-name

Краткое описание (1-2 предложения).

## Роль в экосистеме

**Слой:** [Название слоя] ([N]-й из 5)
**Функция:** Краткое описание функции

→ [Слои согласованности](https://github.com/utemix-lab/extended-mind/blob/main/docs/architecture/COHERENCE_LAYERS.md)

## Описание

...остальное содержимое...
```

### LICENSE

Копируйте MIT лицензию из любого существующего репо:

```
MIT License

Copyright (c) 2024-2026 utemix-lab
...
```

## 3. Ecosystem слой (если применимо)

Для репо, участвующих в экосистемном графе:

```
new-repo/
└── ecosystem/
    ├── ECOSYSTEM_LAYER.md
    └── graph/
        ├── ecosystem_nodes.jsonl
        ├── ecosystem_edges.symbolic.jsonl
        ├── ecosystem_edges.vector.jsonl
        └── inbox/
            └── from-ecosystem.codex.jsonl
```

---

## 4. Регистрация в экосистеме

После создания репо:

### 4.1. Обновить utemix-lab/README.md

Добавить строку в таблицу репозиториев.

### 4.2. Обновить Universe Graph

В `extended-mind/ecosystem/graph/`:

1. Добавить узел в `ecosystem_nodes.jsonl`:
```json
{"id": "new-repo", "type": "Project", "label": "new-repo", "layer": "renderer"}
```

2. Добавить связи в `ecosystem_edges.symbolic.jsonl`:
```json
{"source": "new-repo", "target": "contracts", "type": "reads"}
```

### 4.3. Обновить LLM_PLAYBOOK.md (если значимый репо)

Добавить описание в секцию "Ключевые репозитории".

### 4.4. Добавить в workspace

В `utemix-lab.code-workspace`:
```json
{
  "name": "new-repo",
  "path": "../new-repo"
}
```

## 6. Антипаттерны

### ❌ Не делайте

- Создавать репо без чёткой роли
- Дублировать функциональность существующих репо
- Создавать репо без лицензии
- Игнорировать слои согласованности
- Создавать изолированные репо без связей

### ✅ Делайте

- Сначала проверять, можно ли решить задачу в существующем репо
- Чётко определять слой и функцию
- Документировать связи с экосистемой
- Использовать единую лицензию (MIT)
- Регистрировать в Universe Graph

## Примеры

### Хороший пример: godot-sandbox

- **Слой:** Рендер (5-й)
- **Функция:** Тестовая песочница для контрактов
- **Связи:** Читает contracts, тестирует UI
- **Обоснование:** Нужен отдельный рендер для Godot

### Плохой пример: "utils"

- **Слой:** ? (не определён)
- **Функция:** "Разные утилиты"
- **Связи:** Не определены
- **Проблема:** Нет чёткой роли, лучше распределить по существующим репо
